Random Access Files in JAVA

I'm not used to technical blogging and just want to clarify this blog is mainly to help my understnading of topics and to share what I've learnt so that it may help others :)

So I said I’d do a post every week and I think what’s holding me back from blogging more is:

  • finding/making the time
  • wanting to create really good quality posts
So I would rather put a lot of thought and value into a post than just write for the sake of it. One thing I’ve noticed is that a lot of people LOVE to make things more complicatded than they actually are. So with this in mind, I’ve made a decision to wherever possible try and keep things, including code examples as simple as possible. Today I have decided to write about Random Access Files in Java as this is what I’m currently focusing on right now.

In order to be able to use random access files in Java we need to first import the random access file package: import java.io.RandomAccessFile . This enables us to use the Random Access File class and its associated methods.

The code below basically opens a file,writes data to the file, reads the file length (one of the inbuilt methods) and outputs its result. An important thing to note is when using the length() method we must also catch an IOException otherwise the program will not compile.

input 
import java.io.File;
import java.io.IOException;
import java.io.RandomAccessFile;
import java.io.FileNotFoundException;

public class hello{

public static void main(String[] args){
    try{
    RandomAccessFile raf = new RandomAccessFile(new File("hello.txt"),"rw");
    System.out.println("file opened / read succesfully");
    System.out.println(raf.length());
    raf.writeUTF("hello world!");
    
    }
    catch(IOException e){
        System.out.println("error opening / reading file");
        e.printStackTrace();
    }
    
    
    }
}


As can be seen, when creating the file ‘hello.txt’ we need to specify how we want to use this file, by the second argument, the first is the file name, which is a string. In this example, I have said that we want to be able to read and write, which is signified by the initals ‘rw’. If I only wanted to read I would, as you probably can guess, put only ‘r’ as the second argument.

One of the ways in which we can gain random access to the file is by using the seek() method. The seek method basically lets the user access a specific place in the file, which is determined by the argument passed to the seek() function. The RAF is treated as a binary file and must be handled as a stream of bytes. The file pointer is set to the argument given in the seek() method, which is the position in bytes.

When a file is first opened, the pointer is always set to 0. It’s really important to be familiar with datatypes and there associated memory if we want to navigate succesfully through Random Access Files, for example if an integer is being read, the pointer will be moved by 4 bytes, because this is the size of an integer.

A brief overview of datatypes and their size is shown below:



Again, there are more complex tables available but for this post I just want to keep things simple to enhance clarity and understanding. Another really useful method is .writeUTF() and the way I am using here, it will add the text ‘hello world’ into the file hello.txt. It’s useful to note that when I ran the program without writing the file the System.out.println(raf.length()) command printed 0 but after including the writeUTF(“hello world!”) the value printed is now 14 because this is within the file, as can be seen in the output below.

output
~/workspace/Java/ $ java hello
file opened / read succesfully
14

I hope this post was helpful and informative and if you have any questions please get in touch by using the 'contact' link above :) Please remember if you want to run this code locally then you will need to create a 'hello.txt' file in the same directory for this to work.

VR World Congress 2017

the opportunities are limitless



Wow! I recentley went to the VR World Congress organised by Opposable VR in Bristol and it was absoloutley amazing. It was my first time into the world of VR and it's inspired me so much to start learning image processing such as three.js and webgl, prompting me to explore JavaScript further.

I've been to Bristol a few times in the last few weeks and I absoloutley love the city, prompting me to make a decision to move there. I was working with Microsoft Hololens for the congress, it was easily the busiest stall and we had to ensure everyone queued in a suitable manner! Microsoft Hologram is a Holographic Processing Unit (HPU), a single device which bridges the gap between AR (augmented reality) and VR (virtual reality) and it runs on Windows 10. It was my first foray into both AR and VR. I have to say AR is less intense but no less amazing than VR (which is geared more towards entertainment) - it seems the 'hardcore' gamers will be into VR and AR definitley seems more accesible to the general population.

It's really amazing to see the potential applications of both AR and VR - from healthcare to architecture to education, the possibilities are limitless. During my Hololens demo, I visited space and a few minutes later was given a full on interactive image of the anatomy of the human body - this really has the potential to revolutionize education in an amazing way. The image below is me having a great time exploring the possibilities of the Hololens:



The way in which the Hololens works is that its built in sensors map the environment - this is a huge benefit as it doesn't need external positioning devices. One thing I noticed was that some exhibitors were having difficulties because the floor was uneven and sometimes this would cause difficulty with positioning and tracking the space. The Hololens uses hand gestures, which act as control inputs.

With regards to audio, the Hololens uses a 3D speaker system, which doesn't cover the ears, allowing sounds from the 'real world' to be heard without interference. This is amazing and has great potential.

Yes the device is quite bulky, but the fact that it is a standalone device should also be taken into consideration, with no wiring needed. Something I found absoloutley incredible is the game 'roboraid'. Now I am not a gamer, well not anymore - I used to be a big fan of Spyro! but this game was AMAZING , mainly due to the graphics. The images were so 'beautiful',the graphics mixed with the surfaces of the space around was something that can't be explained in a blog post. This is why I am more drawn to AR than VR because of the possibility of utilising both the space a round and the capability of the AR technology to produce awe inspiring experiences. I feel so inspired to start learning three.js after experience these graphics and even though RoboRaid is developed with unity it's still opened my eyes to the creative nature of programming and the amazing things that can be built.

Another incredible thing I saw at the VRWC was ultrahaptics (a Bristol based company) device that enables users to 'feel' invisible objects. Again, the possibilities of using this technology are so numerous and very very exciting. Imagine being in a fight game and FEELING the force of the opponent!!

During the event, I was lucky enough to try the device, a square 16 * 16 inch pad covered by ultrasound transducers. The sonic 'objects' produced can be felt from up to two meteres away - it's clear this technology has unbelieveable potential!

There were plenty of other things to explore at the conference, which saw over 750 delegates from 20+ countries and it was difficult to experience everything in such a short time. A few days earlier, I'd seen Ghost in the Shell in the cinema and AMD were hosting a VR experience of the film. It was my second time using VR technology and I had to take the headset off it was too intense! The film itself is very intense and the VR experience was literally out of this world! Learning JAVA recap: Applets

the journey continues...

So it's coming to the end of my JAVA 2 module with Cardiff univeristy and mainly this module has focused on Swing applications and GUIs. I created a basic application which caluclated stopping distances, aswell as learning about the popular tile shuffle program. The module began about learning about inheritance, one of the core principles of OOP and then transitioned into focusing on the Swing toolkit and GUIs. I thought this would be a good time to do a recap of what I learnt in the JAVA course so far, with a view of doing more blog posts detailing specific functionality, like super() for example. The first assignment for JAVA1 focused on using nested for loops, something I found familiar as I had practiced a lot of programming before starting the course. The second assignment focused on creating an applet.

So ugly I know and kinda ashamed to have on my blog! ;-) As you can probably guess,this is not one of the aspects I most enjoy about JAVA but nevertheless I learnt alot about applets and their relation to JAVA. This also introduces the idea of inheritence - the class used extends Applet, meaning it inherits all methods and properties and can be used by the class. Different to an implementing an interface, all methods and properties do NOT have to be implemented. The lifecycle of an Applet is as follows, all Applet methods don't take any arguments:
  1. Applet is initalized, using init() method (invoked only once)
  2. Applet is started, using start() method
  3. Applet is painted, using paint() method
  4. Applet is stopped, using stop() method
  5. Applet is destroyed, using destroy() method (invoked only once)
Applets must be used with HTML files, again I have experience with using HTML as HTML and CSS were my first introduction to programming during Dec 2015, when I was based in Portugal - wow time flies and I feel like I've learnt a lot since then! Within the html the user must enter a: <applet code> tag which embeds the applet into the html. The html document also requires the height and width of the application screen. All this is interesting, but you can have more functionality by using GUIs such as by using JFrames and the Swing toolkit, which will be discussed in another blog post. The main thing I learnt that during this module I knew that I wanted to focus more on back-end work that front-end, as may be obvious from the applet ;-). This week I am attending the Virtual Reality World Conference in Bristol so will hopefully do a write up of this on the blog soon too but until then thank you so much for reading! :)
